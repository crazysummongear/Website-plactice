# 設計レビュー

**目的**: 実装前に設計品質を体系的にチェックし、後続の問題を防止する

**対象**: 新機能・バグ修正・リファクタリングの設計ドキュメント

**実行時間**: 30-60分

---

## 前提条件

- `requirements.md` が作成済み
- `design.md` が作成済み
- `.kiro/specs/{feature-name}/` に設定ファイルが存在

---

## フロー

### Phase 1: 要件定義の確認（10分）

#### 1.1 requirements.md の構成確認

**チェックリスト**:
- [ ] 機能要件が明確に定義されているか
- [ ] 非機能要件（パフォーマンス・セキュリティ・可用性）が定義されているか
- [ ] テスト戦略が定義されているか
- [ ] E2E シナリオが列挙されているか
- [ ] Mock 戦略が定義されているか

#### 1.2 ユーザーストーリーの確認

**チェックリスト**:
- [ ] 各要件に「As a [role], I want [feature], so that [benefit]」形式のストーリーがあるか
- [ ] 受け入れ基準が具体的で測定可能か
- [ ] 曖昧な表現（「簡単に」「高速に」など）がないか

**修正例**:
```markdown
❌ 曖昧
- ユーザーが簡単にログインできる

✅ 具体的
- ユーザーがメールアドレスとパスワードを入力して3秒以内にログインできる
- ログイン失敗時にエラーメッセージが表示される
- パスワード忘れ時にリセットメールが送信される
```

#### 1.3 テスト戦略の確認

**チェックリスト**:
- [ ] Unit Test の対象が明確か
- [ ] Component Test の対象が明確か
- [ ] Integration Test の対象が明確か
- [ ] E2E Test のシナリオが具体的か

**テスト戦略テンプレート**:
```markdown
## テスト戦略

### Unit Test
- 対象: ユーティリティ関数、ビジネスロジック
- 例: `validateEmail()`, `calculateTotal()`

### Component Test
- 対象: React コンポーネント
- 例: `LoginForm` のバリデーション表示

### Integration Test
- 対象: API 呼び出し + UI 更新
- 例: ログイン → ダッシュボード表示

### E2E Test
- 対象: ユーザーフロー全体
- 例: サインアップ → メール検証 → ログイン → ダッシュボード
```

---

### Phase 2: 設計の確認（20分）

#### 2.1 design.md の構成確認

**チェックリスト**:
- [ ] アーキテクチャ図が含まれているか
- [ ] 状態管理の設計が明確か
- [ ] テスタビリティ（data-testid）が設計されているか
- [ ] ライブラリ管理が記録されているか
- [ ] E2E 考慮事項が記録されているか
- [ ] アーキテクチャ制約が明示されているか

#### 2.2 状態管理の確認

**チェックリスト**:
- [ ] Single Source of Truth が定義されているか
- [ ] State Owner が明確か
- [ ] Consumer が明確か
- [ ] 状態の複製が禁止されているか

**状態管理テンプレート**:
```markdown
## 状態管理

### Single Source of Truth
- 認証状態: `AuthContext`
- トランザクション: `React Query`

### State Owner
- `AuthProvider` が認証状態を管理
- `QueryClient` がサーバー状態を管理

### Consumer
- `LoginPage` が認証状態を消費
- `DashboardPage` がトランザクション状態を消費

### 禁止事項
- `useAuth` を複数の Hook で複製しない
- `useAuthContext` から取得する
```

#### 2.3 data-testid の設計確認

**チェックリスト**:
- [ ] すべてのインタラクティブ要素に data-testid が設計されているか
- [ ] 命名規則が `{feature}-{element}` か
- [ ] 重複がないか

**data-testid 設計テンプレート**:
```markdown
## data-testid 設計

| 要素 | data-testid | 用途 |
|------|-----------|------|
| ログインボタン | login-button | ログイン実行 |
| メール入力 | email-input | メール入力 |
| パスワード入力 | password-input | パスワード入力 |
| ダッシュボードタイトル | dashboard-title | ページ遷移確認 |
| ボトムナビゲーション | transactions-nav-button | ナビゲーション |
```

#### 2.4 ライブラリ互換性の確認

**チェックリスト**:
- [ ] 採用ライブラリのバージョンが記録されているか
- [ ] 採用理由が記録されているか
- [ ] 互換性確認結果が記録されているか

**ライブラリ管理テンプレート**:
```markdown
## ライブラリ管理

| ライブラリ | バージョン | 採用理由 | 互換性確認 |
|----------|----------|--------|---------|
| React | 18.2.0 | 最新安定版 | ✅ 確認済み |
| React Router | 6.x | SPA ルーティング | ✅ Context API 動作確認 |
| zod | 4.0.0 | スキーマバリデーション | ✅ @hookform/resolvers 互換性確認 |
| @hookform/resolvers | 5.4.0 | React Hook Form 統合 | ✅ zod v4 互換性確認 |
```

#### 2.5 E2E 考慮事項の確認

**チェックリスト**:
- [ ] position: fixed 要素が記録されているか
- [ ] sticky 要素が記録されているか
- [ ] modal/overlay が記録されているか
- [ ] animation が記録されているか
- [ ] loading 状態が記録されているか

**E2E 考慮事項テンプレート**:
```markdown
## E2E 考慮事項

| UI特性 | テストへの影響 | 対処方法 |
|------|------------|--------|
| position: fixed | ビューポート外になる | dispatchEvent('click') |
| sticky | スクロール位置に依存 | スクロール後にクリック |
| modal | 背景要素がクリック不可 | モーダル閉じるまで待機 |
| animation | 要素が安定するまで時間 | waitForAnimations |
| loading | ボタンが disabled | toBeEnabled() で待機 |
```

#### 2.6 アーキテクチャ制約の確認

**チェックリスト**:
- [ ] Provider 配置が明示されているか
- [ ] Hook 利用ルールが明示されているか
- [ ] Singleton ルールが定義されているか

**アーキテクチャ制約テンプレート**:
```markdown
## アーキテクチャ制約

### Provider 配置
```
App.tsx
├── QueryClientProvider
├── AuthProvider
└── BrowserRouter
```

### Hook 利用ルール
- `useAuthContext` は認証が必要なコンポーネントのみ
- `useQuery` は API 呼び出しが必要なコンポーネントのみ
- `useForm` はフォームコンポーネントのみ

### Singleton ルール
- `QueryClient` は単一インスタンス
- `AuthContext` は単一インスタンス
```

### Phase 3: リスク評価（15分）

#### 3.1 リスク項目の抽出

以下のカテゴリでリスクを評価：

| リスク | 重要度 | 対処方法 |
|------|------|--------|
| テスト戦略不足 | High | テスト設計を追加 |
| 状態管理の複製 | High | 設計を修正 |
| ライブラリ互換性未確認 | High | 互換性テスト実施 |
| E2E 考慮事項不足 | Medium | 考慮事項を追加 |
| data-testid 不足 | Medium | data-testid を追加 |

#### 3.2 High リスクの確認

**High リスク項目**:
- [ ] テスト戦略が不完全か
- [ ] 状態管理に複製がないか
- [ ] ライブラリ互換性が確認されているか
- [ ] セキュリティ要件が満たされているか
- [ ] パフォーマンス要件が満たされているか

**High リスク発見時の対処**:
```
❌ High リスク発見
→ 設計を修正
→ 修正後に再レビュー
→ すべての High リスクが解決されるまで実装開始しない
```

#### 3.3 Medium リスクの確認

**Medium リスク項目**:
- [ ] E2E 考慮事項が不足していないか
- [ ] data-testid が不足していないか
- [ ] ドキュメントが不完全でないか

**Medium リスク発見時の対処**:
```
⚠️ Medium リスク発見
→ 設計に追加
→ 実装時に対処
→ テスト時に検証
```

### Phase 4: レビュー結果の記録（5分）

#### 4.1 レビュー結果ドキュメント作成

**ファイル**: `.kiro/specs/{feature-name}/DESIGN_REVIEW.md`

**テンプレート**:
```markdown
# 設計レビュー結果

**実施日**: 2026年5月30日
**レビュアー**: Kiro
**対象**: kakei — 認証機能

## レビュー結果

### 要件定義
- [x] 機能要件が明確
- [x] 非機能要件が定義
- [x] テスト戦略が定義
- [x] E2E シナリオが列挙
- [x] Mock 戦略が定義

### 設計
- [x] アーキテクチャ図が含まれている
- [x] 状態管理が明確
- [x] data-testid が設計されている
- [x] ライブラリ管理が記録されている
- [x] E2E 考慮事項が記録されている
- [x] アーキテクチャ制約が明示されている

### リスク評価

| リスク | 重要度 | 状態 | 対処方法 |
|------|------|------|--------|
| テスト戦略 | High | ✅ 解決 | E2E シナリオを追加 |
| 状態管理 | High | ✅ 解決 | Single Source of Truth を明示 |
| ライブラリ互換性 | High | ✅ 解決 | 互換性テスト実施 |

## 結論

**レビュー結果**: ✅ **承認** - 実装開始可能

## 推奨事項

- [ ] 実装開始前に E2E テスト環境を準備
- [ ] ライブラリ互換性テストを実施
- [ ] 設計ドキュメントを実装チームと共有
```

#### 4.2 レビュー判定

**承認** の場合:
```
✅ 設計レビュー承認
→ 実装開始可能
→ タスク実行開始
```

**修正要求** の場合:
```
❌ 設計レビュー修正要求
→ 設計ドキュメントを修正
→ 修正後に再レビュー
→ 承認されるまで実装開始しない
```

## チェックリスト

実施前：
- [ ] requirements.md が完成しているか
- [ ] design.md が完成しているか
- [ ] 設計ドキュメントを読み込んだか

実施中：
- [ ] すべての確認項目をチェックしたか

## トラブルシューティング

| 問題 | 原因 | 解決方法 |
|------|------|--------|

| リスク多数 | 設計が不十分 | 設計を見直す |
| テスト戦略不足 | テスト設計を忘れた | テスト戦略を追加 |
