## プロジェクト概要

### 🎯 プロジェクト名
**kakei（家計管理アプリ）**

### 📝 目的
- 個人の収支管理をシンプルに行える Web アプリケーション
- AWS クラウドを活用したモダンなアーキテクチャの学習
- AI 駆動開発（Kiro）による効率的な実装

### 👥 対象ユーザー
- 個人利用者（家計管理をしたい人）
- 学習者（モダンなアーキテクチャを学びたい開発者）

## 設計思想

### 1. **シンプルさ優先（KISS 原則）**

```
Keep It Simple, Stupid
```

**理由：**
- MVP（最小限の機能）を素早く実装
- 複雑な機能は後から追加
- 学習コストを最小化

**具体例：**
- DynamoDB の Single Table Design（1 テーブルで完結）
- GSI（Global Secondary Index）を使わない
- Lambda 関数を最小限に抑える

---

### 2. **コスト最適化（月額 $5 以内）**

```
個人利用・学習目的のため、コストを最小限に抑える
```

**理由：**
- 個人プロジェクトで高額な費用は避けたい
- AWS の無料枠を最大限活用（ただし、無料枠は利用不可）
- 必要最小限のリソースで運用

**具体例：**
- DynamoDB: PAY_PER_REQUEST（オンデマンド課金）
- Lambda: メモリ 128MB、タイムアウト 10 秒
- CloudFront: PriceClass_100（北米・ヨーロッパのみ）
- PITR（Point-in-Time Recovery）無効化

---

### 3. **モダンなアーキテクチャ**

```
サーバーレス + Jamstack + Single Table Design
```

**理由：**
- スケーラビリティ（自動スケーリング）
- 運用コストの削減（サーバー管理不要）
- 開発速度の向上（インフラ管理が不要）

**具体例：**
- フロントエンド: React + Vite + Tailwind CSS
- バックエンド: Lambda + API Gateway
- データベース: DynamoDB（NoSQL）
- 認証: Cognito
- インフラ: Terraform（IaC）

---

### 4. **Infrastructure as Code（IaC）**

```
インフラをコードで管理し、再現性を確保
```

**理由：**
- 手作業によるミスを防ぐ
- バージョン管理（Git）で変更履歴を追跡
- チーム開発での共有が容易
- 環境の再現性（dev/staging/prod）

**具体例：**
- Terraform でインフラを定義
- モジュール化（S3、CloudFront、DynamoDB、Cognito など）
- `terraform apply` で自動デプロイ

---

### 5. **AI 駆動開発（Kiro）**

```
AI エージェントを活用した効率的な開発
```

**理由：**
- 開発速度の向上
- ベストプラクティスの自動適用
- 学習コストの削減

**具体例：**
- Spec 駆動開発（要件 → 設計 → タスク → 実装）
- タスクの自動実行（オーケストレーター）
- サブエージェントによる専門特化

## 技術スタック選定理由

### フロントエンド

| 技術 | 選定理由 |
|------|---------|
| **React** | コンポーネントベースで再利用性が高い、エコシステムが豊富 |
| **TypeScript** | 型安全性でバグを減らす、IDE サポートが充実 |
| **Vite** | 高速なビルド、開発体験が良い |
| **Tailwind CSS** | ユーティリティファーストで開発速度が速い |
| **React Hook Form** | フォームバリデーションが簡単 |
| **Zod** | TypeScript と相性が良いスキーマバリデーション |
| **React Query** | サーバー状態管理が簡単、キャッシング機能 |

---

### バックエンド

| 技術 | 選定理由 |
|------|---------|
| **Lambda** | サーバーレス、自動スケーリング、従量課金 |
| **API Gateway** | REST API の簡単な構築、Cognito 連携 |
| **Node.js** | JavaScript/TypeScript で統一、Lambda との相性が良い |
| **DynamoDB** | NoSQL、サーバーレス、自動スケーリング |
| **Cognito** | ユーザー認証・管理が簡単、JWT トークン発行 |

---

### インフラ

| 技術 | 選定理由 |
|------|---------|
| **Terraform** | IaC のデファクトスタンダード、AWS との相性が良い |
| **S3** | 静的ファイルホスティング、低コスト |
| **CloudFront** | CDN、高速配信、HTTPS 対応 |
| **AWS** | クラウドプロバイダーのデファクトスタンダード |

## アーキテクチャ設計の判断基準

### 1. **Single Table Design（DynamoDB）**

#### **選択した理由**

```
✅ コスト削減
   └─ テーブル数が少ない → ストレージコストが低い
   └─ GSI を使わない → インデックスコストが不要

✅ シンプルさ
   └─ 1 テーブルで完結 → 管理が簡単
   └─ JOIN 不要 → クエリが高速

✅ スケーラビリティ
   └─ パーティションキーで自動分散
```

#### **トレードオフ**

```
❌ 複雑なクエリが難しい
   └─ 解決策: アプリケーション側でフィルタリング

❌ データモデルの変更が難しい
   └─ 解決策: MVP 段階では問題なし、将来的に再設計
```

---

### 2. **サーバーレスアーキテクチャ**

#### **選択した理由**

```
✅ 運用コストの削減
   └─ サーバー管理不要
   └─ 自動スケーリング

✅ 従量課金
   └─ 使った分だけ課金
   └─ 低トラフィックなら低コスト

✅ 開発速度
   └─ インフラ管理が不要
   └─ ビジネスロジックに集中
```

#### **トレードオフ**

```
❌ コールドスタート
   └─ 解決策: Lambda のウォームアップ（将来的に検討）

❌ 実行時間制限（15 分）
   └─ 解決策: 長時間処理は Step Functions を使う（将来的に検討）
```

---

### 3. **Cognito による認証**

#### **選択した理由**

```
✅ マネージドサービス
   └─ ユーザー管理が簡単
   └─ セキュリティがデフォルトで高い

✅ JWT トークン発行
   └─ API Gateway との連携が簡単
   └─ ステートレスな認証

✅ メール検証
   └─ スパム防止
   └─ パスワードリセット機能
```

#### **トレードオフ**

```
❌ カスタマイズ性が低い
   └─ 解決策: Lambda トリガーでカスタマイズ（将来的に検討）

❌ UI のカスタマイズが難しい
   └─ 解決策: Hosted UI を使わず、独自 UI を実装
```

---

### 4. **Terraform による IaC**

#### **選択した理由**

```
✅ 再現性
   └─ 同じコードで同じインフラを構築
   └─ 環境の複製が簡単

✅ バージョン管理
   └─ Git で変更履歴を追跡
   └─ レビューが容易

✅ チーム開発
   └─ コードを共有すれば、誰でも同じ環境を構築
```

#### **トレードオフ**

```
❌ 学習コスト
   └─ 解決策: AI（Kiro）が自動生成

❌ 状態管理
   └─ 解決策: S3 バックエンドで状態を管理（将来的に実装）
```

## コスト最適化戦略

### 月額目標: **$5 以内**

### コスト削減施策

| サービス | 施策 | 削減効果 |
|---------|------|---------|
| **DynamoDB** | PAY_PER_REQUEST、PITR 無効化、GSI なし | 約 $0.10/月 |
| **Lambda** | メモリ 128MB、タイムアウト 10 秒 | 約 $0.05/月 |
| **S3** | ライフサイクルポリシー（CSV 7 日後削除） | 約 $0.02/月 |
| **CloudFront** | PriceClass_100（北米・ヨーロッパのみ） | 約 $0.10/月 |
| **Cognito** | 無料枠内（50,000 MAU まで） | $0.00/月 |

**合計: 約 $0.27/月**（目標 $5 以内を大幅に下回る）

## 開発プロセス

### Spec 駆動開発

```
1. 要件定義（requirements.md）
   └─ ユーザーストーリー、受け入れ基準

2. 設計（design.md）
   └─ アーキテクチャ、データモデル、API 設計

3. タスク分解（tasks.md）
   └─ 実装タスクのリスト化

4. 実装（自動実行）
   └─ Kiro がタスクを自動実行

5. レビュー
   └─ ユーザーが確認・フィードバック
```

---

### AI 駆動開発の特徴

#### **1. オーケストレーター + サブエージェント**

```
メインエージェント（オーケストレーター）
    ↓
    ├─ spec-task-execution（タスク実行専門）
    ├─ context-gatherer（コンテキスト収集専門）
    └─ feature-requirements-first-workflow（要件定義専門）
```

#### **2. 同一ワークスペースでの開発**

```
現状: インフラ（Terraform）とアプリ（backend/frontend）を
      同じチャット・ワークスペースで処理

理由:
  ✅ プロジェクト規模が小さい（MVP 段階）
  ✅ 一気通貫の開発が効率的
  ✅ 依存関係の自動解決
  ✅ 学習目的には混在の方が理解しやすい

将来的には:
  🔄 プロジェクトが大きくなったら分離を検討
  🔄 専門エージェント（インフラ専門、アプリ専門）の導入
```

## 将来の拡張性

### PHASE 2: 機能拡張（6 ヶ月）

```
- 予算管理・アラート機能
- カテゴリ別分析
- 家族共有機能
```

### PHASE 3: 本格運用（12 ヶ月）

```
- モバイルアプリ化（React Native）
- AI 機能（支出予測、カテゴリ自動分類）
- 資産管理機能（投資ポートフォリオ、FIRE シミュレーション）
```

---

### 拡張時の考慮事項

#### **1. インフラとアプリの分離**

```
将来的には:
  ├─ インフラチャット（AWS 専門エージェント）
  ├─ バックエンドチャット（Lambda/Node.js 専門エージェント）
  └─ フロントエンドチャット（React 専門エージェント）
```

#### **2. マルチテナント対応**

```
現状: シングルテナント（個人利用）
将来: マルチテナント（家族共有、チーム利用）
```

#### **3. パフォーマンス最適化**

```
- Lambda のウォームアップ
- DynamoDB の GSI 追加（必要に応じて）
- CloudFront のキャッシュ戦略最適化
```

## まとめ

### 設計思想の核心

```
1. シンプルさ優先（KISS 原則）
2. コスト最適化（月額 $5 以内）

```
✅ 学習目的に適しているか

✅ スケーラビリティがあるか
✅ 運用が簡単か
✅ エコシステムが豊富か

```
✅ Spec 駆動開発（要件 → 設計 → タスク → 実装）

```
