## 1. アプリ開発の全体像

### 1.1 アプリの3つの層

すべてのアプリは、大きく3つの層に分かれています：

```
┌─────────────────────────────────────┐
│     フロントエンド（画面）            │
│  ユーザーが見て操作する部分           │
│  例: ボタン、フォーム、グラフ         │
└─────────────────────────────────────┘
              ↕ (通信)
┌─────────────────────────────────────┐
│     バックエンド（サーバー）          │
│  ビジネスロジック、データ処理         │
│  例: 計算、検証、保存                 │
└─────────────────────────────────────┘
              ↕ (読み書き)
┌─────────────────────────────────────┐
│     データベース（データ保管庫）      │
│  すべてのデータを永続保存             │
│  例: ユーザー情報、取引履歴           │
└─────────────────────────────────────┘
```

### 1.2 kakeiの例

```
┌──────────────────────────────────────┐
│  React（フロント）                    │
│  - ログイン画面                       │
│  - 収支入力フォーム                   │
│  - ダッシュボード                     │
└──────────────────────────────────────┘
           ↕ HTTPS
┌──────────────────────────────────────┐
│  Lambda（バック）                     │
│  - ユーザー認証                       │
│  - 収支の保存・取得                   │
│  - CSV解析                           │
└──────────────────────────────────────┘
           ↕ 読み書き
┌──────────────────────────────────────┐
│  DynamoDB（DB）                       │
│  - ユーザーデータ                     │
│  - 取引履歴                           │
│  - カテゴリ情報                       │
└──────────────────────────────────────┘
```

## 2. バックエンド vs フロントエンド

### 2.1 バックエンド（サーバー側）

**役割**: ビジネスロジックとデータ管理

| 項目 | 説明 |
|------|------|
| **言語** | Node.js, Python, Java, Go など |
| **主な仕事** | データ検証、計算、保存、認証 |
| **実行場所** | クラウド（AWS, GCP など） |
| **ユーザーから見える？** | 見えない（裏側） |
| **セキュリティ** | 重要（秘密鍵、DB接続情報を保管） |

**kakeiの例**:
```typescript
// Lambda関数（バックエンド）
export async function createTransaction(event) {
  // 1. ユーザーを認証
  const userId = getUserId(event);
  
  // 2. データを検証
  if (!body.amount || body.amount <= 0) {
    return errorResponse('Invalid amount');
  }
  
  // 3. DynamoDBに保存
  await put({
    PK: `USER#${userId}`,
    SK: `TX#${date}#${id}`,
    amount: body.amount,
    // ...
  });
  
  // 4. 結果を返す
  return successResponse(transaction);
}
```

### 2.2 フロントエンド（クライアント側）

**役割**: ユーザーインターフェース

| 項目 | 説明 |
|------|------|
| **言語** | JavaScript, TypeScript, React など |
| **主な仕事** | UI表示、ユーザー入力、バックエンド呼び出し |
| **実行場所** | ユーザーのブラウザ |
| **ユーザーから見える？** | 見える（全部） |
| **セキュリティ** | 低い（ユーザーが見られる） |

**kakeiの例**:
```typescript
// React コンポーネント（フロントエンド）
export function TransactionForm() {
  const [amount, setAmount] = useState('');
  
  const handleSubmit = async () => {
    // 1. ユーザー入力を取得
    const data = {
      date: '2026-05-26',
      amount: parseInt(amount),
      category: 'food',
    };
    
    // 2. バックエンドに送信
    const response = await fetch('/transactions', {
      method: 'POST',
      body: JSON.stringify(data),
    });
    
    // 3. 結果を画面に表示
    if (response.ok) {
      alert('保存しました！');
    }
  };
  
  return (
    <form onSubmit={handleSubmit}>
      <input 
        value={amount}
        onChange={(e) => setAmount(e.target.value)}
      />
      <button type="submit">保存</button>
    </form>
  );
}
```

### 2.3 通信の流れ

```
ユーザーが「保存」ボタンをクリック
         ↓
フロント: 入力値を取得
         ↓
フロント: バックエンドにHTTPリクエスト送信
         ↓
バック: リクエストを受け取る
         ↓
バック: データを検証
         ↓
バック: DynamoDBに保存
         ↓
バック: 結果をレスポンスで返す
         ↓
フロント: レスポンスを受け取る
         ↓
フロント: 画面を更新（「保存しました」と表示）
```

## 3. 開発の流れ（3つのアプローチ）

### 3.1 アプローチ1: バックエンド優先

```
バックエンド完成 → フロントエンド作成
```

**メリット**:
- ✅ APIが安定してからフロント開発できる
- ✅ バックエンドのバグが先に見つかる
- ✅ チーム開発に向いている

**デメリット**:
- ❌ フロント開発が遅れる
- ❌ バックエンドの設計ミスに気づきにくい

**使う場面**:
- 複数チーム（バック班、フロント班）で分業
- APIの仕様が明確
- 大規模プロジェクト

**kakeiはこれ！**

### 3.2 アプローチ2: フロントエンド優先

```
画面設計 → バックエンド作成
```

**メリット**:
- ✅ ユーザー体験から考えるので、UXが良い
- ✅ 必要なAPI仕様が明確になる
- ✅ 早期にユーザーフィードバックが得られる

**デメリット**:
- ❌ バックエンドが後付けになる
- ❌ 設計がズレることがある

**使う場面**:
- スタートアップ（ユーザー体験重視）
- 1人開発
- プロトタイプ開発

### 3.3 アプローチ3: 並行開発

```
バックエンド ↔ フロントエンド（同時進行）
```

**メリット**:
- ✅ 早く完成する
- ✅ 両側の問題を早期に発見できる

**デメリット**:
- ❌ 調整が大変
- ❌ 両方の知識が必要

**使う場面**:
- 経験豊富なチーム
- 小規模プロジェクト

### 3.4 どれを選ぶ？

| 状況 | おすすめ | 理由 |
|------|---------|------|
| チーム開発 | バック優先 | 分業しやすい |
| 1人開発 | 並行 or フロント優先 | 柔軟に対応できる |
| スタートアップ | フロント優先 | UX重視 |
| 大規模 | バック優先 | 安定性重視 |
| 学習目的 | 並行 | 両方学べる |

## 4. API設計の重要性

### 4.1 APIとは？

**API** = Application Programming Interface（アプリケーション間の通信ルール）

フロントエンドとバックエンドが「どうやって通信するか」を決めたもの。

### 4.2 APIの3要素

#### 1️⃣ エンドポイント（どこに送るか）

```
GET /transactions          # 収支一覧を取得
POST /transactions         # 新しい収支を作成
PUT /transactions/{id}     # 収支を更新
DELETE /transactions/{id}  # 収支を削除
```

#### 2️⃣ リクエスト（何を送るか）

```json
{
  "date": "2026-05-26",
  "category": "food",
  "amount": 1500,
  "incomeExpense": "EXPENSE",
  "memo": "スーパーで買い物"
}
```

#### 3️⃣ レスポンス（何が返ってくるか）

```json
{
  "id": "tx-001",
  "userId": "user-123",
  "date": "2026-05-26",
  "category": "food",
  "amount": 1500,
  "incomeExpense": "EXPENSE",
  "memo": "スーパーで買い物",
  "createdAt": "2026-05-26T10:30:00Z",
  "updatedAt": "2026-05-26T10:30:00Z"
}
```

### 4.3 APIが明確な利点

APIが明確に決まっていると：

| 利点 | 説明 |
|------|------|
| **並行開発可能** | バック班とフロント班が同時に開発できる |
| **テストしやすい** | 期待値が明確なので、テストが書きやすい |
| **変更に強い** | 仕様変更時に影響範囲が明確 |
| **ドキュメント化** | APIドキュメントが自動生成できる |
| **統合テスト簡単** | バックとフロントの結合テストが簡単 |

### 4.4 kakeiのAPI設計

kakeiでは、Design.mdに詳細に記載：

```markdown
## 3. API 設計

### 3.1 エンドポイント一覧

#### 収支管理

| Method | Path | 説明 | 認証 |
|--------|------|------|------|
| GET | `/transactions` | 収支一覧取得 | JWT |
| POST | `/transactions` | 収支登録 | JWT |
| PUT | `/transactions/{id}` | 収支更新 | JWT |
| DELETE | `/transactions/{id}` | 収支削除 | JWT |
```

**これにより**:
- ✅ バックエンド開発者は「何を実装するか」が明確
- ✅ フロントエンド開発者は「何を呼び出すか」が明確
- ✅ 両者が同時に開発できる

## 5. kakeiプロジェクトの開発順序

### 5.1 全体の流れ

```
Wave 1-4: インフラ + バックエンド
    ↓
Wave 5-7: 認証機能（バック + フロント）
    ↓
Wave 8-10: 収支管理（フロント作成中）
```

### 5.2 各Waveの役割

| Wave | 内容 | 理由 |
|------|------|------|
| 1-4 | インフラ + バックエンド | APIが必要だから |
| 5-7 | 認証機能 | ユーザー管理が必須だから |
| 8-10 | フロントエンド | APIが完成してから |

### 5.3 なぜこの順序？

```
フロントエンドは、バックエンドなしでは動かない
    ↓
だから、バックエンドを先に作る
    ↓
バックエンドは、インフラなしでは動かない
    ↓
だから、インフラを先に作る
```

**つまり：下から上へ積み上げる**

```
フロントエンド（画面）← 最後に作る
    ↑
バックエンド（API）← 中盤に作る
    ↑
インフラ（AWS）← 最初に作る
```

## 6. ドキュメント駆動開発

### 6.1 ドキュメントの役割

kakeiでは、以下のドキュメントが開発を導いています：

| ドキュメント | 役割 | 例 |
|-------------|------|-----|
| **Requirements.md** | 「何を作るか」を定義 | 「ユーザーが収支を入力できる」 |
| **Design.md** | 「どう作るか」を定義 | 「DynamoDBのスキーマ」「APIエンドポイント」 |
| **Tasks.md** | 「何をするか」を定義 | 「5.1.1 型定義ファイルを作成」 |

### 6.2 ドキュメント駆動開発のメリット

```
Requirements.md（要件）
    ↓
Design.md（設計）
    ↓
Tasks.md（実装タスク）
    ↓
コード実装
```

**メリット**:
- ✅ 実装前に設計を確認できる
- ✅ チーム全体が同じ理解を共有できる
- ✅ 後から「なぜこう作ったのか」が分かる
- ✅ 新しいメンバーがキャッチアップしやすい

### 6.3 kakeiの例

**Requirements.md に書かれている**:

```
入力項目: 日付、カテゴリ、金額、収入/支出区分、メモ

```

**Design.md に書かれている**:

## 7. テスト駆動開発

### 7.1 テストの3層

```
┌─────────────────────────────────────┐
│  E2E テスト（エンドツーエンド）      │
│  ユーザーの操作全体をテスト          │
│  例: ログイン → 収支入力 → 確認      │
└─────────────────────────────────────┘
         ↑
┌─────────────────────────────────────┐
│  統合テスト（インテグレーション）    │
│  複数のコンポーネント間の連携        │
│  例: フロント + バック + DB          │
└─────────────────────────────────────┘
         ↑
┌─────────────────────────────────────┐
│  単体テスト（ユニット）              │
│  1つの関数・コンポーネントをテスト   │
│  例: TransactionForm の入力検証      │
└─────────────────────────────────────┘
```

### 7.2 kakeiのテスト

**既に実装されているテスト**:
- ✅ PrivateRoute の単体テスト
- ✅ PrivateRoute の統合テスト

**まだ必要なテスト**:
- ❌ LoginPage のバリデーションテスト
- ❌ TransactionForm のテスト
- ❌ Lambda ハンドラーのテスト

### 7.3 テストの重要性

| テスト | 重要度 | 理由 |
|--------|--------|------|
| 単体テスト | 🔴 高 | バグを早期に発見 |
| 統合テスト | 🟡 中 | 連携ミスを発見 |
| E2E テスト | 🟡 中 | ユーザー視点でテスト |

## 8. よくある質問

### Q1: バックエンドとフロントエンド、どっちが難しい？

**A**: 状況による

| 観点 | バックエンド | フロントエンド |
|------|-------------|--------------|
| **ロジック** | 複雑 | シンプル |
| **UI/UX** | 不要 | 重要 |
| **パフォーマンス** | 重要 | 重要 |
| **セキュリティ** | 🔴 超重要 | 低い |
| **デバッグ** | 難しい | 簡単 |

**結論**: 両方難しい。バックエンドはロジック、フロントエンドはUX。

---

### Q2: APIドキュメントはどうやって作る？

**A**: 3つの方法

#### 1️⃣ 手書き（kakeiの方法）
```markdown
## API エンドポイント

GET /transactions
- 説明: 収支一覧取得
- リクエスト: なし
- レスポンス: { items: [...] }
```

**メリット**: 柔軟、カスタマイズ可能  
**デメリット**: 手間がかかる

#### 2️⃣ OpenAPI/Swagger
```yaml
openapi: 3.0.0
paths:
  /transactions:
    get:
      summary: 収支一覧取得
      responses:
        200:
          description: Success
```

**メリット**: 自動生成、ツール連携  
**デメリット**: 学習コスト

#### 3️⃣ コードから自動生成
```typescript
/**
 * @route GET /transactions
 * @returns {Transaction[]} 収支一覧
 */
export function getTransactions() { ... }
```

**メリット**: 常に最新  
**デメリット**: ツール依存

### Q3: 開発中にAPIの仕様が変わったら？

**A**: 以下の手順で対応

```
1. Design.md を更新
2. Tasks.md を更新
3. バックエンドコードを修正
4. フロントエンドコードを修正
5. テストを修正
6. 統合テストで確認
```

**重要**: ドキュメント → コード の順序を守る

### Q4: 1人開発の場合、どのアプローチがいい？

**A**: 「並行開発」がおすすめ

**理由**:
- バックエンドを少し作る
- フロントエンドを少し作る
- 統合テストで確認
- 問題があれば修正
- 繰り返す

**メリット**:
- ✅ 早期に問題を発見できる
- ✅ 両方の知識が深まる
- ✅ 柔軟に対応できる

---

### Q5: テストはいつ書く？

**A**: 3つのタイミング

#### 1️⃣ テスト駆動開発（TDD）
```
テスト作成 → コード実装 → テスト実行
```
**メリット**: 品質が高い  
**デメリット**: 時間がかかる

#### 2️⃣ コード実装後
```
コード実装 → テスト作成 → テスト実行
```
**メリット**: 実装が早い  
**デメリット**: テスト漏れがある

#### 3️⃣ 本番前
```
コード実装 → 本番前にテスト → 本番リリース
```
**メリット**: 最小限のテスト  
**デメリット**: バグが本番で見つかる

**kakeiの場合**: 2番目（実装後にテスト）
