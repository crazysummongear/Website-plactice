## 1. テストの基本的な考え方

### 1.1 テストピラミッド

```
        /\
       /  \
      / E2E \     ← 少ない（遅い、壊れやすい）
     /--------\
    /  統合    \   ← 中くらい
   /------------\
  /   単体      \  ← 多い（速い、安定）
 /----------------\
```

**初心者向けの理解**:
- **単体テスト**: 1つの関数やコンポーネントだけをテスト（例: ボタンを押したら何が起こるか）
- **統合テスト**: 複数のコンポーネントが連携して動くかをテスト（例: ログイン → ダッシュボード表示）
- **E2E テスト**: 実際のユーザーの操作をシミュレート（例: ブラウザを開いて実際にクリック）

---

### 1.2 何をテストすべきか？（初心者向けガイド）

#### ✅ テストすべきもの
1. **ユーザーが触る部分**
   - ボタンをクリックしたら何が起こるか
   - フォームに入力したらどうなるか
   - エラーメッセージが表示されるか

2. **お金や個人情報に関わる部分**
   - 金額の計算が正しいか
   - 認証が正しく動くか
   - データが正しく保存されるか

3. **壊れたら困る部分**
   - ログインできなくなる
   - データが消える
   - アプリがクラッシュする

#### ❌ テストしなくてもいいもの
1. **ライブラリの動作**
   - React Router が動くか（React Router のテストチームが既にテスト済み）
   - Cognito が動くか（AWS がテスト済み）

2. **見た目の細かい調整**
   - ボタンの色が正しいか
   - フォントサイズが正しいか
   - （これは手動で確認する方が早い）

3. **定数や設定値**
   - `const MAX_LENGTH = 100` みたいな値

## 3. テストが足りていない部分

### 3.1 フロントエンド（React コンポーネント）

#### 🔴 優先度：高（すぐにテストすべき）

##### 1. **LoginPage のバリデーション**
**何が未検証か**:
- メールアドレスが空欄の場合、エラーメッセージが表示されるか？
- パスワードが12文字未満の場合、エラーメッセージが表示されるか？
- 無効なメールアドレス（例: `test@`）を入力した場合、エラーが出るか？

**なぜ重要か**:
- ユーザーが間違った入力をしたときに、適切なフィードバックがないと混乱する
- セキュリティ上、パスワードの強度チェックは必須

**テスト例**:
```typescript
it('メールアドレスが空欄の場合、エラーメッセージが表示される', async () => {
  render(<LoginPage />);
  
  const submitButton = screen.getByRole('button', { name: /ログイン/ });
  await userEvent.click(submitButton);
  
  expect(screen.getByText('メールアドレスを入力してください')).toBeInTheDocument();
});
```

---

##### 2. **SignupPage のバリデーション**
**何が未検証か**:
- パスワードと確認用パスワードが一致しない場合、エラーが出るか？
- パスワードが弱い場合（大文字・小文字・数字・特殊文字を含まない）、エラーが出るか？
- メール検証コードが6桁でない場合、エラーが出るか？

**なぜ重要か**:
- サインアップは一度しか行わないが、失敗すると大きなストレス
- パスワードの強度チェックはセキュリティの基本

---

##### 3. **AuthContext のエラーハンドリング**
**何が未検証か**:
- ログインに失敗した場合（間違ったパスワード）、エラーメッセージが表示されるか？
- ネットワークエラーが発生した場合、適切なエラーメッセージが表示されるか？
- セッションが期限切れになった場合、ログアウトされるか？

**なぜ重要か**:
- エラーハンドリングがないと、ユーザーは「何が起こったのか」わからない
- 「ログインできない」という問い合わせが増える

**テスト例**:
```typescript
it('ログインに失敗した場合、エラーメッセージが表示される', async () => {
  // Cognito の login をモックして、エラーを返す
  vi.mocked(authHook.useAuth).mockReturnValue({
    login: vi.fn().mockRejectedValue(new Error('Incorrect username or password')),
    error: 'ユーザー名またはパスワードが正しくありません',
    // ...
  });
  
  render(<LoginPage />);
  
  // ログインを試みる
  await userEvent.type(screen.getByLabelText('メールアドレス'), 'test@example.com');
  await userEvent.type(screen.getByLabelText('パスワード'), 'wrongpassword');
  await userEvent.click(screen.getByRole('button', { name: /ログイン/ }));
  
  // エラーメッセージが表示される
  expect(screen.getByText('ユーザー名またはパスワードが正しくありません')).toBeInTheDocument();
});
```

### 3.2 バックエンド（Lambda 関数）

#### 🔴 優先度：高

##### 7. **transactions ハンドラのバリデーション**
**何が未検証か**:
- 金額が負の数の場合、エラーが返されるか？
- 日付が未来の日付の場合、エラーが返されるか？
- 必須フィールド（amount, date, type）が欠けている場合、エラーが返されるか？

**なぜ重要か**:
- 金額の計算ミスは致命的
- 不正なデータが保存されると、後で修正が大変

**テスト例**:
```typescript
it('金額が負の数の場合、400エラーが返される', async () => {
  const event = {
    body: JSON.stringify({
      amount: -100,  // 負の数
      date: '2024-01-01',
      type: 'expense',
      category: '食費',
    }),
    // ...
  };
  
  const response = await handler(event);
  
  expect(response.statusCode).toBe(400);
  expect(JSON.parse(response.body).message).toContain('金額は正の数である必要があります');
});
```

##### 8. **transactions ハンドラの認証チェック**
**何が未検証か**:
- JWT トークンがない場合、401エラーが返されるか？
- 無効な JWT トークンの場合、401エラーが返されるか？
- 他のユーザーのデータにアクセスしようとした場合、403エラーが返されるか？

**なぜ重要か**:
- セキュリティの基本
- 他人のデータが見えてしまうのは致命的

---

##### 9. **DynamoDB への保存・取得**
**何が未検証か**:
- データが正しく DynamoDB に保存されるか？
- 保存したデータが正しく取得できるか？
- 更新・削除が正しく動作するか？

**なぜ重要か**:
- データが保存されないと、アプリの意味がない
- データが消えると、ユーザーの信頼を失う

---

#### 🟡 優先度：中

##### 10. **CSV インポート機能**
**何が未検証か**:

- 大量のデータ（1000行以上）をインポートした場合、タイムアウトしないか？

## 4. 優先順位の付け方

### 4.1 優先順位の判断基準

| 基準 | 説明 | 例 |
|------|------|-----|
| **影響範囲** | 壊れたら何人のユーザーに影響するか | ログイン機能 → 全ユーザー（高） |
| **頻度** | どれくらいの頻度で使われるか | ログイン → 毎日（高）、CSV インポート → 月1回（低） |
| **リスク** | 壊れたらどれくらい困るか | データ消失 → 致命的（高）、見た目の崩れ → 軽微（低） |
| **複雑さ** | テストを書くのにどれくらい時間がかかるか | 単純なバリデーション → 簡単（高）、E2E → 複雑（低） |

---

### 4.2 優先順位マトリックス

```
影響大 ┌─────────────┬─────────────┐
      │  🔴 最優先   │  🟡 重要    │
      │  すぐやる    │  余裕あれば  │
      ├─────────────┼─────────────┤
      │  🟡 重要    │  🟢 低優先  │
影響小 │  余裕あれば  │  後回し     │
      └─────────────┴─────────────┘
        頻度低         頻度高
```

**現在のプロジェクトに当てはめると**:
- 🔴 **最優先**: LoginPage のバリデーション、transactions ハンドラのバリデーション
- 🟡 **重要**: SignupPage のバリデーション、Dashboard のテスト
- 🟢 **低優先**: CSV インポート、Terraform のテスト

## 6. テストを書く際のチェックリスト

### 6.1 テストを書く前に確認すること

- [ ] このテストは「ユーザーが触る部分」をテストしているか？
- [ ] このテストは「壊れたら困る部分」をテストしているか？
- [ ] このテストは「ライブラリの動作」ではなく「自分のコード」をテストしているか？
- [ ] このテストは「見た目」ではなく「動作」をテストしているか？

### 6.2 テストを書いた後に確認すること

- [ ] テストが失敗する場合、何が原因かすぐにわかるか？
- [ ] テストが成功した場合、本当に正しく動作していると言えるか？
- [ ] テストが壊れやすくないか？（ちょっとした変更で失敗しないか？）
- [ ] テストが速く実行されるか？（1秒以内が理想）
