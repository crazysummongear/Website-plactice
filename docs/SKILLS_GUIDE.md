## スキル一覧

### 1. E2E テスト修正フロー

**ファイル**: `~/.kiro/skills/e2e-test-fix-flow.md`

**目的**: Playwright E2E テスト失敗時の根本原因分析と修正を体系的に実施する

**対象**: React + Playwright を使用したフロントエンドプロジェクト

**実行時間**: 30-60分

**使用場面**:
- E2E テストが失敗した
- テストが不安定（時々失敗する）
- UI 変更後にテストが失敗した

**主な内容**:
- Phase 1: テスト実行と失敗分析
- Phase 2: 要素表示タイムアウト問題の分析
- Phase 3: 要素表示位置の問題分析
- Phase 4: ロジック・状態エラーの分析
- Phase 5: ページ遷移失敗の分析

**実行手順**:
```bash
# 1. スキルを開く
code ~/.kiro/skills/e2e-test-fix-flow.md

# 2. Phase 1 から順に実行
npm run test:e2e

# 3. 失敗パターンに応じて対応する Phase に進む
```

**期待される成果**:
- ✅ すべての E2E テストが通過
- ✅ テスト失敗の根本原因が特定される
- ✅ 修正内容がコミットされる

### 2. ライブラリ互換性テスト

**ファイル**: `~/.kiro/skills/library-compatibility-test.md`

**目的**: メジャーバージョンアップ前にライブラリ間の互換性を確認する

**対象**: React エコシステムのライブラリ（特に zod + @hookform/resolvers、React + React Router など）

**実行時間**: 20-40分

**使用場面**:
- ライブラリのメジャーバージョンアップを検討している
- `npm outdated` で赤色（Major）のライブラリがある
- 新しいライブラリを採用する前に互換性を確認したい

**主な内容**:
- Phase 1: 互換性確認対象の特定
- Phase 2: 互換性テスト環境の準備
- Phase 3: 互換性テスト実施
  - zod + @hookform/resolvers の互換性テスト
  - React + React Router の互換性テスト
  - @tanstack/react-query + React の互換性テスト
  - 実装コードでの動作確認
- Phase 4: 互換性テスト結果の記録

**実行手順**:
```bash
# 1. スキルを開く
code ~/.kiro/skills/library-compatibility-test.md

# 2. Phase 1 で確認対象を特定
npm outdated

# 3. Phase 2-3 でテストを実施
npm install zod@latest
node -e "..."  # テストコマンド実行

# 4. Phase 4 で結果を記録
```

**期待される成果**:
- ✅ ライブラリ間の互換性が確認される
- ✅ アップグレード可否の判定が下される
- ✅ テスト結果がドキュメント化される

### 3. 設計レビュー

**ファイル**: `~/.kiro/skills/design-review.md`

**目的**: 実装前に設計品質を体系的にチェックし、後続の問題を防止する

**対象**: 新機能・バグ修正・リファクタリングの設計ドキュメント

**実行時間**: 30-60分

**使用場面**:
- 新機能の設計が完了した
- 実装を開始する前に設計品質を確認したい
- 設計に不足がないか確認したい

**主な内容**:
- Phase 1: 要件定義の確認
- Phase 2: 設計の確認
  - アーキテクチャ図
  - 状態管理
  - data-testid 設計
  - ライブラリ互換性
  - E2E 考慮事項
  - アーキテクチャ制約
- Phase 3: リスク評価
- Phase 4: レビュー結果の記録

**実行手順**:
```bash
# 1. スキルを開く
code ~/.kiro/skills/design-review.md

# 2. requirements.md と design.md を読み込む
code .kiro/specs/{feature-name}/requirements.md
code .kiro/specs/{feature-name}/design.md

# 3. Phase 1-3 のチェックリストを実行
# 4. Phase 4 でレビュー結果をドキュメント化
```

**期待される成果**:
- ✅ 設計品質が確認される
- ✅ リスク項目が特定される
- ✅ 実装開始の判定が下される

---

### 4. エラー根本原因分析

**ファイル**: `~/.kiro/skills/error-root-cause-analysis.md`

**目的**: 問題発生時に根本原因を体系的に特定し、再発防止策を立案する

**対象**: テスト失敗、ビルドエラー、ランタイムエラー、ロジックエラー

**実行時間**: 30-90分（問題の複雑さによる）

**使用場面**:
- テストが失敗した
- ビルドエラーが発生した
- アプリケーションがクラッシュした
- 期待と異なる結果が返された

**主な内容**:
- Phase 1: エラー情報の収集
- Phase 2: ビルドエラーの分析
- Phase 3: テスト失敗の分析
- Phase 4: ランタイムエラーの分析
- Phase 5: ロジックエラーの分析
- Phase 6: 環境エラーの分析
- Phase 7: 再発防止策の立案

**実行手順**:
```bash
# 1. スキルを開く
code ~/.kiro/skills/error-root-cause-analysis.md

# 2. Phase 1 でエラー情報を収集
# 3. エラーの種類に応じて対応する Phase に進む
# 4. Phase 7 で再発防止策を立案
```

**期待される成果**:
- ✅ 根本原因が特定される
- ✅ 修正内容が実装される
- ✅ 再発防止策が立案される
- ✅ エラー分析レポートが作成される

## スキルの使用方法

### 方法 1: スキルファイルを直接開く

```bash
# スキルファイルを開く
code ~/.kiro/skills/e2e-test-fix-flow.md

# 手順に従って実行
```

### 方法 2: Kiro コマンドで読み込む

Kiro IDE で以下のコマンドを実行：
```
Kiro: Activate Skill
```

スキル名を選択：
- `e2e-test-fix-flow`
- `library-compatibility-test`
- `design-review`
- `error-root-cause-analysis`

### 方法 3: チャットで参照

Kiro チャットで以下のように質問：
```
E2E テストが失敗しました。修正方法を教えてください。
```

Kiro が自動的に `e2e-test-fix-flow` スキルを参照して対応します。

## スキル間の関連性

```
新機能開発フロー:
1. 設計レビュー → 設計品質を確認
2. ライブラリ互換性テスト → 新しいライブラリの互換性を確認
3. 実装開始
4. E2E テスト修正フロー → テスト失敗時に対応
5. エラー根本原因分析 → 問題発生時に対応

バグ修正フロー:
1. エラー根本原因分析 → 根本原因を特定
2. 設計レビュー → 修正設計を確認
3. 実装開始
4. E2E テスト修正フロー → テスト失敗時に対応
```

## スキル活用のベストプラクティス

### 1. 予防的な使用

**設計レビュー** を実装前に必ず実施：
- 設計品質を確認
- リスク項目を特定
- 後続の問題を防止

### 2. 計画的なアップグレード

**ライブラリ互換性テスト** でメジャーバージョンアップを検証：
- 互換性を事前確認
- 予期しない問題を防止
- 安全にアップグレード

### 3. 迅速な問題解決

**エラー根本原因分析** で問題を体系的に分析：
- 根本原因を特定
- 修正内容を検証
- 再発防止策を立案

### 4. テスト品質の維持

**E2E テスト修正フロー** でテスト失敗に対応：
- 失敗パターンを分類
- 対応方法を体系化
- テスト品質を維持

## スキル改善のフィードバック

スキルを使用して改善点を発見した場合：

1. **改善内容を記録**
   ```markdown
   # スキル改善提案
   
   **スキル名**: E2E テスト修正フロー
   **改善点**: Phase 2 に「要素の visibility 確認」を追加
   **理由**: position: fixed 以外の visibility 問題も多い
   ```

2. **ドキュメントを更新**
   ```bash
   code ~/.kiro/skills/e2e-test-fix-flow.md
   # 改善内容を追加
   ```

3. **チームに共有**
   - スキルの改善内容を共有
   - 他のプロジェクトでも活用
